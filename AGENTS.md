# AGENTS.md - Development Guidelines for Charlotte Wailly Website

This document provides essential information for AI coding agents working on the Charlotte Wailly website project.

## Project Overview

This is a static website for Charlotte Wailly, a lawyer and mediator specializing in family law, urban planning, and construction law. The site is built with vanilla HTML, CSS/SCSS, and JavaScript using the PicoCSS framework.

## Build Commands

### Primary Build Commands
- **`./entrypoint.sh build`** - Build the site by copying source files to the `dist/` directory
- **`./entrypoint.sh open`** - Open the built site in the default browser
- **`./entrypoint.sh deploy`** - Build and deploy to GitHub Pages
- **`./entrypoint.sh cname`** - Generate CNAME file for domain configuration

### Manual Build Process
The build process copies files from `src/` to `dist/`:
```bash
rm -rf ./dist
mkdir ./dist
cp -r ./src/css ./dist/css
cp -r ./src/favicon ./dist/favicon
cp -r ./src/images ./dist/images
cp -r ./src/js ./dist/js
cp ./src/charlotte_wailly_cv.pdf ./dist/charlotte_wailly_cv.pdf
cp ./src/index.html ./dist/index.html
```

### Testing Commands
Since this is a static website without automated tests:
- **Manual testing**: Open `dist/index.html` in browser after build
- **Cross-browser testing**: Test in Chrome, Firefox, Safari, Edge
- **Mobile testing**: Test responsive design on various screen sizes
- **Accessibility testing**: Use browser dev tools accessibility checker

### Single Test Execution
No automated test framework is used. For manual testing of specific features:
1. Build the site: `./entrypoint.sh build`
2. Open in browser: `./entrypoint.sh open`
3. Test specific functionality manually

## Code Style Guidelines

### HTML Structure
- Use semantic HTML5 elements (`<header>`, `<nav>`, `<main>`, `<section>`, `<article>`, `<footer>`)
- Include proper ARIA attributes for accessibility
- Use French language (`lang="fr"`)
- Follow PicoCSS class naming conventions
- Maintain proper heading hierarchy (h1 → h2 → h3)
- Use data attributes for JavaScript hooks (`data-theme`, etc.)

### JavaScript Style
- Use modern ES6+ syntax (const/let, arrow functions, template literals)
- Follow functional programming principles where appropriate
- Use descriptive variable names in camelCase
- Add comments to separate logical sections with `// --------------------------------------------------`
- Handle DOM manipulation safely with proper null checks
- Use event delegation when possible
- Maintain separation of concerns (theme toggle, scroll behavior, map initialization)

### CSS/SCSS Style
- Use SCSS syntax with nesting (max 3 levels deep)
- Follow PicoCSS design system and CSS custom properties
- Use CSS custom properties for theming (`--pico-primary`, `--tango-spacing-*`)
- Maintain consistent spacing using custom properties
- Use semantic class names (`.logo`, `.icon`, `.container`)
- Avoid deep nesting; prefer single-level selectors when possible
- Comment sections with `/* section name */`

### File Organization
- **`src/`**: Source files
  - `index.html`: Main HTML file
  - `js/main.js`: JavaScript functionality
  - `scss/styles.scss`: Main stylesheet
  - `css/styles.css`: Compiled CSS (generated)
  - `images/`: Image assets
  - `favicon/`: Favicon files
  - `charlotte_wailly_cv.pdf`: CV document
- **`dist/`**: Built files (generated by build process)

### Imports and Dependencies
- **External Libraries**:
  - PicoCSS: `https://cdn.jsdelivr.net/npm/@picocss/pico@2/css/pico.cyan.min.css`
  - Leaflet.js: `https://unpkg.com/leaflet@1.9.4/dist/leaflet.js`
  - Google Fonts: Jost font family
- **Local Files**: Use relative paths (`./css/styles.css`, `./js/main.js`)
- No build system or bundler; files are copied as-is

### Naming Conventions

#### Files and Directories
- Use lowercase with underscores: `charlotte_wailly_cv.pdf`
- Group related files in directories: `images/`, `favicon/`, `js/`, `css/`

#### CSS Classes
- Use kebab-case for CSS classes: `.is-fixed-above-lg`, `.theme-toggle`
- Follow PicoCSS conventions: `.container`, semantic classes
- Use BEM-like naming for components: `.logo`, `.icon`

#### JavaScript
- camelCase for variables and functions: `themeToggle`, `scrollPosition`
- Use const for constants and let for variables
- Descriptive names: `html`, `themeIcon`, `sections`

#### HTML Attributes
- Use data attributes for JS hooks: `data-theme="light"`
- Semantic ARIA attributes: `role="banner"`, `role="navigation"`
- Standard HTML attributes: `lang="fr"`, `charset="utf-8"`

### Error Handling
- JavaScript: Use try-catch blocks for external API calls (Leaflet map)
- Graceful degradation: Ensure site works without JavaScript
- Validate user inputs where applicable
- Handle missing DOM elements safely

### Security Best Practices
- Use HTTPS for external resources (CDNs)
- Include integrity checks for CDN resources
- Avoid inline JavaScript; keep scripts in separate files

### Performance Considerations
- Optimize images before adding to repository
- Minimize HTTP requests (few external resources)
- Use CSS for animations instead of JavaScript where possible
- Lazy load images if adding more content
- Minify CSS/JS if build process expands

### Git Workflow
- Commit built files in `dist/` (they are part of the deployment)
- Use conventional commit messages in French where appropriate
- Deploy via GitHub Pages using `git subtree push`

### Accessibility Guidelines
- Include alt text for all images
- Use semantic HTML structure
- Ensure sufficient color contrast
- Support keyboard navigation
- Include proper meta tags and descriptions
- Test with screen readers

### Browser Support
- Modern browsers: Chrome, Firefox, Safari, Edge
- Mobile browsers: iOS Safari, Chrome Mobile
- Graceful degradation for older browsers

## Development Workflow

1. Make changes to source files in `src/`
2. Run `./entrypoint.sh build` to generate dist files
3. Test locally with `./entrypoint.sh open`
4. Commit changes (including dist files)
5. Deploy with `./entrypoint.sh deploy` when ready
